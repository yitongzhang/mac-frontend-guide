04

AppKit vs SwiftUI

AppKit is imperative (like vanilla DOM manipulation). SwiftUI is declarative (like React). SwiftUI is mature enough for most apps in 2025.

AppKit (imperative)
// Manual state management func updateLabel() { label.stringValue = "Count: \(count)" } func increment() { count += 1 updateLabel() }
SwiftUI (declarative)
// Automatic UI updates @State var count = 0 Text("Count: \(count)") Button("+") { count += 1 }

body = computed property returning the UI

State: @State is useState

@State creates reactive state. When it changes, the view re-renders.

struct Counter: View { @State private var count = 0 var body: some View { Button("Count: \(count)") { count += 1 } } }
React equivalent
function Counter() { const [count, setCount] = useState(0) return ( <button onClick={() => setCount(c => c + 1)}> Count: {count} </button> ) }

@State = property wrapper for local state

private = only this view owns it

Mutating count triggers re-render automatically

Styling: The Mental Model

CSS applies styles via selectors. SwiftUI applies them inline via modifiers. Same goal, different approach.

CSS/Tailwind
<!-- Styles live separately --> <div class="p-4 bg-blue-500 rounded-lg"> Hello </div> /* Or in CSS */ .card { padding: 16px; background: blue; border-radius: 8px; }
SwiftUI
// Styles are inline modifiers Text("Hello") .padding(16) .background(Color.blue) .cornerRadius(8)
Key Differences
1.
No cascade — modifiers apply directly to the view, no inheritance surprises
2.
Order matters — .padding().background() ‚â† .background().padding()
3.
Type-safe — compiler catches typos (no "colour" bugs)
4.
Composable — extract modifiers into reusable ViewModifiers

Reusable Styles with ViewModifier

Like Tailwind's @apply or CSS classes. Bundle modifiers into reusable units.

Tailwind @apply
/* Define once */ .btn-primary { @apply px-4 py-2 bg-blue-500 text-white rounded-lg font-semibold; } <!-- Use anywhere --> <button class="btn-primary"> Save </button>
SwiftUI ViewModifier
// Define once struct PrimaryButton: ViewModifier { func body(content: Content) -> some View { content .padding(.horizontal, 16) .padding(.vertical, 8) .background(Color.blue) .foregroundColor(.white) .cornerRadius(8) .fontWeight(.semibold) } } // Use anywhere Button("Save") { ... } .modifier(PrimaryButton())
// Pro tip: Create an extension for cleaner syntax extension View { func primaryButtonStyle() -> some View { modifier(PrimaryButton()) } } // Now use it like a native modifier Button("Save") { ... } .primaryButtonStyle()

Conditional Styling (Hover, Pressed, Disabled)

Like :hover, :active, :disabled in CSS. SwiftUI uses state in ButtonStyle or direct conditionals.

CSS pseudo-classes
.button { background: blue; } .button:hover { background: darkblue; } .button:active { transform: scale(0.98); } .button:disabled { opacity: 0.5; }
SwiftUI ButtonStyle
struct MyButtonStyle: ButtonStyle { func makeBody(configuration: Config) -> some View { configuration.label .background( configuration.isPressed ? Color.blue.opacity(0.8) : Color.blue ) .scaleEffect( configuration.isPressed ? 0.98 : 1.0 ) } }
// For hover (macOS), use .onHover modifier @State private var isHovered = false Text("Hover me") .foregroundColor(isHovered ? .blue : .primary) .onHover { hovering in isHovered = hovering } // For disabled state Button("Submit") { } .disabled(isLoading) // grays out automatically

Theming with @Environment

Like React Context or CSS custom properties cascading down. Inject values through the view hierarchy.

React Context
// Provider at top <ThemeContext.Provider value={theme}> <App /> </ThemeContext.Provider> // Consume anywhere below const theme = useContext(ThemeContext)
SwiftUI @Environment
// Inject at top ContentView() .environment(\.myTheme, customTheme) // Read anywhere below @Environment(\.myTheme) var theme
// Define custom environment key struct ThemeKey: EnvironmentKey { static let defaultValue = Theme.light } extension EnvironmentValues { var myTheme: Theme { get { self[ThemeKey.self] } set { self[ThemeKey.self] = newValue } } } // Built-in environment values you can read: @Environment(\.colorScheme) var colorScheme // .light or .dark @Environment(\.sizeCategory) var sizeCategory // Dynamic Type size @Environment(\.locale) var locale // User's locale

Responsive Design

Like media queries, but simpler. Use GeometryReader for sizes, @Environment for traits.

CSS media queries
.grid { display: grid; grid-template-columns: 1fr; } @media (min-width: 768px) { .grid { grid-template-columns: 1fr 1fr; } }
SwiftUI GeometryReader
GeometryReader { geo in if geo.size.width > 600 { HStack { Sidebar() Content() } } else { VStack { Content() } } }
// Or use horizontal size class for iPad/Mac vs iPhone @Environment(\.horizontalSizeClass) var sizeClass var body: some View { if sizeClass == .compact { // iPhone-style layout TabView { ... } } else { // iPad/Mac sidebar layout NavigationSplitView { ... } } }
‚úì
SwiftUI adapts automatically in many cases. NavigationStack, Form, and system controls adjust for device size without manual breakpoints.

Animations & Transitions

Like CSS transitions, but declarative. Just say what should animate, SwiftUI handles the rest.

CSS transitions

SwiftUI animations
